      Three, system overview

      ORB-SLAM3 is a system based on ORB-SLAM2 and ORB-SLAM-VI. It can run robustly in pure visual or visual inertial navigation systems (monocular, binocular and RGB-D using pinholes or fisheyes) Model, you can also define your own model).

      • Atlas is a hybrid map composed of a series of discrete maps. An active map will be maintained here to locate new key frames, and the local mapping thread will continuously optimize and update this map. There are also non-active maps in Atlas. This system encodes the system based on the bag of words model for relocation, closed-loop detection and map fusion.
      • The tracking thread processes sensor information, calculates the pose of the current frame and the active map in real time, and minimizes the reprojection error of matching feature points. This thread also has a key frame screening process. In the VIO mode, the speed of the body and the bias of the IMU are optimized by using the variability of the inertial navigation. When the tracking thread is lost, the tracking thread tries to complete the relocation of the current frame in all Alats. If the stock relocation is successful, the tracking is pulled back, otherwise the activate will be stored as non-active after a period of time. A new active map will be initialized.
      • The local mapping thread adds key frames and points to the active map, deletes redundant frames, and uses visual or visual inertial navigation BA to optimize the map. These are all done in a local sliding window. In addition, in the case of inertial navigation, the parameters of the IMU are initialized and optimized using MAP estimation.
      • The closed-loop detection and map fusion threads detect the same area in the active map and the entire Atlas. If the same area is active, the closed loop process will be executed. If it belongs to different maps, they will be merged into one map. After the closed-loop correction, a global BA is triggered in a thread to optimize the map.

      Fourth, the camera model

      Provides pinhole model and fisheye model. The camera model is separately abstracted in the system (re-projection and back-projection equations, Jacobian equations, etc.). This allows us to use our own camera model in the system. In the process of abstraction, we also encountered some difficulties, which will be discussed below.

      A. Relocation

      ORB-SLAM uses Epnp to solve relocation; but it is based on a calibrated pinhole camera model. In order to continue our work, we use the camera-independent ML-pnp algorithm to completely decouple from the camera model. Because he uses projection light as input. The camera model only needs to provide a back-projection function that is passed from the pixels to the projection light in order to be able to use relocation.

      B. Uncorrected binocular SLAM

      Many SLAM systems assume that the binocular camera has been corrected (both images are converted to pinhole projections with the same focal length, the image planes are coplanar, and aligned with the horizontal epipolar line, so that by observing the same image in the other image OK, you can easily match the features in one of the images). However, the assumption of correcting stereo images has great limitations and is not suitable or feasible in many applications. For example, correcting a fisheye camera requires severe cropping. This loses the advantages of a wide-angle camera with a large viewing angle, rapid map creation, and robust occlusion processing.
      Therefore, the system in this article does not rely on image correction, but regards the binocular camera as two monocular cameras:

      • Has a constant SE(3) transformation
      • Two images have partially identical observations

      These constraints are very efficient when we triangulate new landmark points and perform BA optimization. SLAM estimates the pose of a rigid body with 6 degrees of freedom, and its reference frame can be located in one of the cameras or in the IMU sensor. The camera is represented according to the pose of the rigid body. We can use the binocular to initialize the landmark point we saw for the first time. There are many overlapping areas for the binocular, we use it as a monocular camera.

      Visual Inertial Navigation SLAM

      ORB-SLAM-VI is the first visual inertial navigation system capable of map reuse, but it can only be based on a single-purpose pinhole model, which is very slow to initialize. In this article, the system uses fast and accurate IMU initialization and passed a Open source SLAM libraryUse pinhole or fisheye model to quickly complete the initialization of monocular inertial navigation or binocular inertial navigation

      B. IMU initialization

      The purpose of initialization is to provide good initial values for the inertial navigation variables: body speed, gravity direction, and IMU bias. VI-DSO tries not to initialize, and directly uses BA to optimize, resulting in initialization up to 30s.
      In the work of this article, we propose a fast and accurate initialization method based on three key insights:

      • Pure monocular SLAM can provide a very accurate initial map, but the scale is not known. So you can join Imu to get scale information.
      • Scale drift-aware large scale monocular SLAM proves that when the scale is explicitly expressed as an optimization variable, instead of using the implicit representation of BA, it converges faster.
      • Ignoring the uncertainty of the sensor during the IMU initialization process will produce large unpredictable errors (Fast and robust initialization for visual-inertial SLAM_2019ICRA)

      C. Tracking and mapping

      Tracking and mapping use the scheme in Visual-inertial monocular SLAM with map reuse.

      • The tracking thread only optimizes the state of the latest two frames and the position of the map point remains unchanged.
      • The mapping uses the sliding window of key frames and their points as optimizable variables, including key frames that can be viewed together, but keep them fixed.

      In some cases, when slow motion cannot provide good inertial parameter observation capabilities, initialization may not converge to an accurate solution within 15 seconds. In order to make the system more robust, this paper proposes aNew scale optimization methodThis method is based on an improved single-inertial navigation optimization method, in which all key frames are inserted, but scale and gravity direction are the only estimated parameters (Figure 2d). In this case, it is an incorrect assumption that biases remain unchanged. The estimated value of each frame can be used to correct biases. The calculation efficiency of this optimization is very high. It is executed in the local mapping thread every 10 seconds until the mapping exceeds 100 key frames, or the initialization exceeds 75 seconds.

      D. Robust improvement of tracking loss

      The VIO system of this article enters the state of visual tracking failure when the system tracks less than 15 points, and then executes:

      • Short-term failure: use the IMU reading to estimate the pose, project the map points onto the estimated camera pose, and then do the matching in a large image window, and the matching result is included in the VIO optimization. In most cases, visual tracking can be restored, but if it exceeds 5 seconds, it has not been restored. Go to the next state.

      • Long-term failure: re-initialize the visual inertial navigation to build a map, and this map becomes the active map.

      Map fusion and closed loop detection

      The short-term and mid-term data associations established between the image frame and the active map are used in the tracking and mapping thread to project the map points onto the estimated pose, and then match in a small window to obtain the matching relationship. For long-term data association for relocation and closed-loop detection, ORB-SLAM is a bag of words model.
      Unlike tracking, location recognition uses DBoW2 to construct a key frame database using its bag of words vector, and a given query image can effectively provide the most similar key frame according to its bag of words. Using only the first candidate frame, the original DBoW2 query can achieve 50-80% accuracy and recall. In order to prevent false positive observations, DBoW2 implements time and geometric consistency checks to increase the accuracy of the operating point to 100% and the recall rate to 30-40%. It is crucial that the time consistency check delays the position recognition during at least 3 key frames. When trying to use it in our Atlas system, we found that this delay and low recall rates are often caused in repeated areas of the same or different maps.
      In the work of this article, we propose a new scene re-recognition algorithm that improves the recall rate when it is associated with mixed map data for a long time. When the mapping thread filters out a key frame, the scene re-recognition tries to detect the key frame in Atlas for matching. If the matched key frame is in the active map, this determines a closed loop; otherwise, it is the data association of the hybrid map, and the fusion of the matched map and the current active map is performed. The second feature of this method is that once the poses of the current frame and the matched map frame are estimated, we build a partial window between the matched frame and its adjacent frames in the common view. In this window, we focus on searching for interim data association, thereby improving the accuracy of closed-loop detection and map fusion.

      D. Closed loop detection

      Closed-loop detection is similar to map fusion, but the two key frames of scene re-identification are both under the active map. The connection window is formed according to the matched key frames, and the repeated points are detected and merged, and then new edges are constructed in the common view and the essential graph. Then optimize the pose map to propagate the results of the closed-loop correction to the remaining maps. The last step is the global BA, and the MAP estimate is obtained after considering the mid-term and long-term matching of closed-loop detection. In the case of visual inertial navigation, it is only performed when the number of key frames is lower than the threshold to avoid huge computational costs.
